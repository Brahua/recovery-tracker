# Session Handoff - 2026-07-16

## Current Milestone

Recovery Ritual has completed its MVP and UX redesign implementation. The next milestone is a one-week field trial from 2026-07-16 through 2026-07-23 using the hosted `recovery-tracker-staging` project.

The purpose of this window is to collect evidence from normal daily use before adding more functionality. Fix blocking defects when they appear, but defer non-blocking product expansion until the observations can be reviewed together.

## Delivered State

- Separate authenticated routes for `Hoy`, `Registrar`, `Insights`, and `Reporte`.
- Responsive authenticated shell plus a deliberate signed-out landing experience.
- Session and nightly closeout forms backed by Supabase Auth, PostgreSQL, and RLS.
- Dedicated success states for saved sessions and completed days.
- Data-driven insights and medical report surfaces with truthful sparse-data states.
- Functional 7-, 14-, and 30-day report windows, selectable appointment questions, print/PDF output, and share fallback.
- All 17 selected Claude Design artifacts implemented with deliberate deviations recorded in `docs/design/claude-design-reference.md`.
- Current-week strip ordered Monday through Sunday with `Hoy` placed on the actual weekday.
- Recovery-day comparisons normalized to `America/Lima` so evening activity does not roll into the next UTC date.
- Focused unit coverage for view models, calculations, validation, callback errors, and registration state.

## Current Data Behavior

- Multiple rehabilitation sessions are allowed on the same day. Every save creates a separate `rehab_sessions` record.
- A session can contain multiple canonical exercises plus one custom exercise from the current form.
- `occurred_at` represents when the session happened. It defaults to the time the registration form is opened, is editable to minute precision, and is stored as an ISO timestamp.
- `created_at` is generated by PostgreSQL and represents the actual database insertion time.
- Business-day grouping uses `America/Lima`; full persisted timestamps remain UTC-compatible.
- Nightly closeout is limited to one record per user and calendar date.
- `Hoy` treats the session ritual as complete when at least one session exists for the current day.
- When multiple sessions exist, some compact recaps use the most recent session while Insights and Reporte include all records in their selected windows.
- `Registrar` shows compact summaries for the latest two sessions and closeouts. There is no dedicated calendar or full day-detail history yet.

## Field-Trial Questions

Observe these points during normal use:

1. Does the session form feel fast enough immediately after therapy?
2. Is the distinction between session time and save time intuitive?
3. Are multiple same-day sessions represented clearly in `Hoy`, Insights, and Reporte?
4. Does the Monday-to-Sunday strip remain correct around the week boundary?
5. Is one nightly closeout per day the right constraint?
6. Do the completion states lead naturally to the next action?
7. Are the available recent summaries sufficient, or is detailed day-by-day history necessary?
8. Do Insights and Reporte become useful with a week of real data, and are their statements medically conservative?
9. Are there mobile layout, keyboard, focus, loading, or error states that interrupt daily use?

For each observation, record:

- date and approximate time
- route and action being performed
- expected behavior
- observed behavior
- whether it blocked the ritual or was only friction
- screenshot or exact copy when useful
